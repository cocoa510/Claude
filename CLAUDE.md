# 共通開発ルール

> **最優先プロジェクト: ATLAS**（FX戦略自動生成システム）— 詳細は `ATLAS/CLAUDE.md` を参照

## 基本方針

- OS: Windows 11 / Shell: bash（Unixパス記法を使用）
- 言語: Python 3.11+
- コメント・ドキュメント・コミットメッセージ: **日本語**
- コード内の変数名・関数名・クラス名: **英語（snake_case / PascalCase）**

## コーディング規約

- 型ヒントを必ず付ける（`from __future__ import annotations` を活用）
- Pydanticによる入出力バリデーションを徹底する
- 非同期処理は `asyncio` を基本とし、blocking I/Oを避ける
- 例外は握りつぶさず、必ずログに記録してから再raise or 適切な処理をする
- マジックナンバーは定数として定義する

## セキュリティ

- APIキー・パスワード等の秘密情報はコードに直書きしない（`.env` + `python-dotenv`）
- `.env` ファイルは必ず `.gitignore` に含める
- ユーザー入力・外部APIレスポンスは必ず検証する

## Git運用ルール

- コミットメッセージ形式: `[種別] 内容の概要`
  - 種別例: `feat`, `fix:bug`, `change:spec`, `refactor`, `test`, `docs`, `chore`
  - `[fix:bug]` — 既存定義に対する実装バグの修正（出力の期待値は変わらない）
  - `[change:spec]` — 計算ロジック・閾値・インターフェースの意図的な定義変更
  - `[refactor]` — 出力結果を変えない内部構造の整理
- 機能単位でブランチを切る（`feature/xxx`, `fix/xxx`）
- 秘密情報を含むファイルは絶対にcommitしない

## テスト方針

- `pytest` + `pytest-asyncio` を使用
- 外部APIはモックを使いユニットテストを書く
- 本番・Live系の処理には必ず統合テストを用意する

## 戦略の再現性管理

- 生成した戦略は必ず**戦略仕様書**（人間が読める形式）を併せて出力する
- 仕様書にはエントリー/エグジット条件、使用インジケータ、パラメータを明記する
- システム消失時でもFX経験者が手動で再現可能なレベルの記述を保つ
- 戦略コードと仕様書は同一の `generation_id` で紐付けて管理する

## ファイル・ディレクトリ

- 各システムは独立したサブディレクトリで管理
- 各システムディレクトリに独自の `CLAUDE.md` を配置し、システム固有ルールを記載
